# Phase 1: การวิเคราะห์ระบบ (System Analysis)
## บทที่ 3 — การออกแบบและพัฒนาระบบจัดการกุญแจ (KMS)

---

## 1.1 ภาพรวมการวิเคราะห์

ในขั้นตอนแรกของการพัฒนาด้วยกระบวนการ Agile ทีมพัฒนาได้ทำการวิเคราะห์ความต้องการของระบบอย่างละเอียด โดยเริ่มตั้งแต่การกำหนดบทบาทผู้ใช้งาน การออกแบบฐานข้อมูล ไปจนถึงการวางแผนขั้นตอนการทำงานของแต่ละฟังก์ชัน เพื่อให้ระบบสามารถตอบสนองความต้องการของผู้ใช้งานได้อย่างครบถ้วน

ระบบจัดการกุญแจ (KMS) ได้รับการออกแบบสำหรับผู้ใช้งาน 3 กลุ่มหลัก ได้แก่ **นักศึกษา** ที่ต้องการยืมกุญแจห้องเรียน **อาจารย์/เจ้าหน้าที่** ที่มีสิทธิ์พิเศษในการจัดการกุญแจ และ **ผู้ดูแลระบบ (Admin)** ที่ควบคุมการทำงานของระบบทั้งหมด

## 1.2 แผนภาพ Entity Relationship Diagram (ERD)

> 📎 **อ้างอิงไฟล์แผนภาพ**: [`er_diagram.md`](./er_diagram.md)

ERD ของระบบ KMS ประกอบด้วย Entity หลักทั้งหมด 11 ตาราง ดังนี้

| Entity | หน้าที่ |
|---|---|
| **USER** | เก็บข้อมูลผู้ใช้งาน เช่น รหัสนักศึกษา, อีเมล, บทบาท, และคะแนนพฤติกรรม |
| **KEY** | เก็บข้อมูลกุญแจแต่ละดอก พร้อมรหัสห้อง, หมายเลขช่อง, และ NFC Tag UID |
| **BOOKING** | บันทึกรายการเบิก-คืนกุญแจทุกครั้ง รวมถึงสถานะ, เวลา, และคะแนนหักที่เกิดขึ้น |
| **DAILY_AUTHORIZATION** | ตาราง "ต้นฉบับความจริง" ที่ระบุว่าใครมีสิทธิ์เบิกกุญแจห้องไหนในเวลาใด (ต่อวัน) |
| **SCHEDULE** | ตารางสอนประจำสัปดาห์ ระบุวิชา, ห้อง, วัน, และช่วงเวลาสอน |
| **SUBJECT** | รายชื่อวิชาเรียนในระบบ |
| **PENALTY_LOG** | บันทึกการหักคะแนนเมื่อมีการคืนกุญแจล่าช้า |
| **SYSTEM_LOG** | บันทึก action ทุกอย่างที่เกิดขึ้นในระบบพร้อม IP Address |
| **SECTION / MAJOR** | ข้อมูลกลุ่มเรียนและสาขาวิชาของนักศึกษา |
| **PENALTY_CONFIG** | การตั้งค่าระบบหักคะแนน เช่น ระยะเวลาผ่อนผัน, จำนวนคะแนนต่อรอบ |
| **BORROW_REASON** | รายการเหตุผลที่อนุญาตให้เบิกกุญแจนอกตาราง |

### ความสัมพันธ์สำคัญของระบบ

- **SCHEDULE → DAILY_AUTHORIZATION**: ระบบจะแปลงตารางสอนประจำสัปดาห์ให้เป็น DailyAuthorization ทุกวัน เพื่อระบุ "สิทธิ์ของวันนั้น" อย่างชัดเจน
- **BOOKING → PENALTY_LOG**: ทุกครั้งที่มีการคืนกุญแจช้าเกินกว่าเวลาที่กำหนด ระบบจะสร้างบันทึกการหักคะแนนโดยอัตโนมัติ

## 1.3 แผนภาพ Use Case Diagram

> 📎 **อ้างอิงไฟล์แผนภาพ**: [`usecase_diagram.md`](./usecase_diagram.md)

ระบบแบ่งความสามารถตามบทบาทผู้ใช้งาน ดังนี้

### บทบาทนักศึกษา (Student)
นักศึกษาสามารถเบิกกุญแจตามตารางเรียน หรือนอกตารางโดยระบุเหตุผล, คืนกุญแจ, สลับสิทธิ์กุญแจกับผู้อื่น (Swap), และดูประวัติการเบิก-คืนย้อนหลัง

### บทบาทอาจารย์ (Teacher)
อาจารย์มีสิทธิ์เบิกกุญแจได้ทันทีโดยไม่ต้องระบุเหตุผล, โอนสิทธิ์กุญแจให้นักศึกษาหรือบุคคลอื่น, และจัดการตารางสอนของตนเองผ่านระบบเว็บ

### บทบาทเจ้าหน้าที่/Admin
ผู้ดูแลระบบมีสิทธิ์สูงสุด ครอบคลุมการจัดการข้อมูลผู้ใช้ทั้งหมด, การจัดการกุญแจ, การกำหนดตารางเรียน, การดู System Log, และการอนุมัติกรณีพิเศษ

## 1.4 แผนภาพ Sequence Diagram

> 📎 **อ้างอิงไฟล์แผนภาพ**: [`sequence_diagrams.md`](./sequence_diagrams.md)

ทีมพัฒนาได้จัดทำ Sequence Diagram สำหรับฟังก์ชันหลักทั้งหมด ดังนี้

### 1.4.1 การเบิกกุญแจ (Borrow Key)

ขั้นตอนการเบิกกุญแจประกอบด้วย 4 ส่วนหลักที่ทำงานร่วมกัน:
1. **ส่วนผู้ใช้งาน**: เลือกห้อง → สแกนบัตรยืนยันตัวตน → ยืนยันการเบิก
2. **ส่วน Backend**: ตรวจสอบสิทธิ์จาก DailyAuthorization → สร้าง Booking
3. **ส่วน Hardware**: รับคำสั่งปลดล็อก Solenoid → ตรวจจับการดึงกุญแจออกผ่าน NFC
4. **ส่วนความปลอดภัย**: หากไม่ดึงกุญแจออกภายใน 10 วินาที ระบบจะล็อกกลับและยกเลิก Booking โดยอัตโนมัติ

### 1.4.2 การคืนกุญแจ (Return Key)

การคืนกุญแจใช้การตรวจจับ NFC แบบ Real-time ผ่าน Socket.IO:
1. ผู้ใช้สแกนบัตรยืนยันตัวตน
2. ระบบแสดงหน้ารอเสียบกุญแจ (WaitForKeyReturnPage) พร้อมนาฬิกาถอยหลัง 60 วินาที
3. เมื่อเสียบกุญแจในช่อง ESP8266 จะตรวจจับ NFC Tag และส่งสัญญาณกลับมา
4. หากช่องถูกต้อง ระบบจะปิดรายการเบิกและคำนวณ penalty หากคืนช้า

### 1.4.3 การสลับสิทธิ์กุญแจ (Swap Authorization)

ฟังก์ชัน Swap ออกแบบสำหรับ 2 คนที่ต้องการแลกเปลี่ยนกุญแจห้องกัน โดยไม่ต้องนำกุญแจมาคืนที่ตู้ก่อน ระบบจะตรวจสอบว่าทั้งสองคนยังถืออยู่หรือมีสิทธิ์ คาบเรียนไม่ทับซ้อน จึงจะอนุญาตให้สลับสิทธิ์ได้

### 1.4.4 การโอนสิทธิ์กุญแจ (Transfer Authorization)

ฟังก์ชัน Transfer ออกแบบสำหรับกรณีที่ผู้มีกุญแจต้องการมอบความรับผิดชอบให้ผู้อื่น ระบบจะเปลี่ยนชื่อผู้รับผิดชอบใน Booking ปัจจุบัน และบันทึกประวัติการโอนลงใน SystemLog

### 1.4.5 การทำงานของเจ้าหน้าที่และอาจารย์ผ่านระบบเว็บ

เจ้าหน้าที่และอาจารย์สามารถเข้าสู่ระบบผ่าน Web Dashboard ที่พัฒนาด้วย Next.js เพื่อดู Dashboard แบบ Real-time ผ่าน Socket.IO, จัดการตารางสอน และตรวจสอบประวัติการเบิก-คืนย้อนหลัง

## 1.5 สรุปการวิเคราะห์ระบบ

จากการวิเคราะห์พบว่าระบบ KMS มีความซับซ้อนในการประสานงานระหว่างองค์ประกอบ 4 ส่วน ได้แก่ ระบบ Frontend (Kiosk UI), Backend API, Hardware Service, และอุปกรณ์ ESP8266 การออกแบบ DailyAuthorization เป็นจุดกลางในการจัดการสิทธิ์ถือเป็นหัวใจสำคัญของระบบ ทำให้สามารถรองรับทั้งการเบิกตามตารางและการเบิกกรณีพิเศษได้อย่างยืดหยุ่น
