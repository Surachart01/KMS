# Phase 4: การประเมินระบบ (System Evaluation)
## บทที่ 3 — การออกแบบและพัฒนาระบบจัดการกุญแจ (KMS)

---

## 4.1 ภาพรวมการประเมินระบบ

การประเมินระบบ KMS แบ่งออกเป็น **6 ส่วน** โดยอ้างอิงตามขอบเขตโครงงาน (บทที่ 1 ข้อ 1.3) และฟังก์ชันการทำงานจริงของระบบ มีจุดประสงค์เพื่อยืนยันว่าระบบทำงานได้ตามที่ออกแบบไว้ครบถ้วน ทั้งในส่วนของ Hardware, Software (Kiosk), Web Application, ข้อมูล, ประสิทธิภาพ, และฟีเจอร์เพิ่มเติม

**แบบประเมินที่ใช้**: `docs/แบบประเมิน_Test_Cases_V5.docx` (74 Test Cases รวมทั้งหมด)

## 4.2 ส่วนที่ 1: การทดสอบด้านฮาร์ดแวร์ (Hardware Testing)

**อ้างอิงขอบเขต 1.3.1** (ระบบ Hardware ของตู้กุญแจอัตโนมัติ)

### 4.2.1 จุดประสงค์
ทดสอบว่าอุปกรณ์ฮาร์ดแวร์ทุกชิ้นทำงานได้ถูกต้องและสื่อสารกันได้ตามที่ออกแบบไว้ ครอบคลุม NFC Reader, Relay, Solenoid, LED, และ ESP8266 Boards

### 4.2.2 หัวข้อการทดสอบ

| หัวข้อ | ขอบเขตที่อ้างอิง | จุดประสงค์ |
|---|---|---|
| การอ่านบัตร NFC (MIFARE/สมาร์ตโฟน/iOS) | 1.3.1.1 | ตรวจสอบว่าระบบสามารถอ่าน UID ได้จากอุปกรณ์หลายประเภทภายใน 2 วินาที |
| การปฏิเสธบัตรที่ไม่ได้ลงทะเบียน | 1.3.1.1 | ระบบต้องไม่อนุญาตให้เบิกด้วย NFC ที่ไม่รู้จัก |
| การควบคุม Solenoid Lock | 1.3.1.2, 1.3.1.4 | Solenoid ต้องปลดล็อกเมื่อเบิก, ล็อกอัตโนมัติเมื่อดึงกุญแจ หรือหมดเวลา |
| ระบบไฟ LED แสดงสถานะ | 1.3.1.3 | LED ต้องเปลี่ยนสีถูกต้องตามสถานะของแต่ละช่อง |
| ระบบ ESP8266 Multi-Board | 1.3.1.5 | บอร์ดทั้ง 3 ต้องถูกตรวจจับและ Auto-reconnect เมื่อหลุด |

**จำนวน Test Cases**: 16 กรณีทดสอบ (HW-01 ถึง HW-16)

## 4.3 ส่วนที่ 2: การทดสอบซอฟต์แวร์ — ระบบเบิก-คืนกุญแจ

**อ้างอิงขอบเขต 1.3.2.1 ข)** (ระบบ Kiosk)

### 4.3.1 จุดประสงค์
ทดสอบ Business Logic หลักของระบบ ครอบคลุมทุกกรณีที่อาจเกิดขึ้นในการเบิก-คืนกุญแจ รวมถึงการป้องกันข้อผิดพลาดต่างๆ

### 4.3.2 หัวข้อการทดสอบ

| หัวข้อ | จุดประสงค์ |
|---|---|
| เบิกกุญแจตามตาราง (มีสิทธิ์) | ระบบต้องปลดล็อกทันทีโดยไม่ต้องกรอกเหตุผล |
| เบิกกุญแจนอกตาราง (ต้องใส่เหตุผล) | ระบบต้องแสดงฟอร์มให้กรอกเหตุผลก่อนเบิก |
| เบิกกุญแจล่วงหน้า 30 นาที | ระบบต้องอนุญาตการเบิกล่วงหน้าตาม Policy |
| ปฏิเสธเบิกเมื่อมีกุญแจค้างอยู่ | ป้องกันการเบิกซ้ำโดยยังไม่คืนกุญแจเดิม |
| ปฏิเสธเมื่อบัญชีถูกระงับ (Ban) | ผู้ใช้ที่คะแนนหมด (0 คะแนน) ต้องไม่สามารถเบิกได้ |
| ป้องกันคาบชน (Schedule Reserved) | ห้องที่มีคนมีเรียนอยู่ต้องถูกสงวนสิทธิ์ตลอดคาบ |
| ยกเลิกเมื่อไม่ดึงกุญแจภายใน 10 วินาที | ป้องกัน Booking ค้างโดยไม่มีกุญแจถูกดึงออก |
| การคืนกุญแจปกติ / คืนช้า | ตรวจสอบการคำนวณ penalty เมื่อคืนช้าเกินเวลา |
| ป้องกันการเสียบผิดช่อง | แจ้งเตือนหากเสียบกุญแจในช่องที่ไม่ถูกต้อง |
| Auto-Return เมื่อเสียบกุญแจโดยไม่กดเมนู | ระบบตรวจจับ NFC และปิด Booking อัตโนมัติ |
| ระบบหักคะแนน (Penalty) | ตรวจสอบการหักคะแนนและ Ban ผู้ใช้เมื่อคะแนนเหลือ 0 |

**จำนวน Test Cases**: 16 กรณีทดสอบ (SW-01 ถึง SW-16)

## 4.4 ส่วนที่ 3: การทดสอบเว็บแอปพลิเคชัน

**อ้างอิงขอบเขต 1.3.2.1, 1.3.2.2** (ระบบ Web Dashboard)

### 4.4.1 จุดประสงค์
ทดสอบฟังก์ชันการจัดการระบบผ่านเว็บแอปพลิเคชัน ที่ใช้โดยเจ้าหน้าที่และอาจารย์

### 4.4.2 หัวข้อการทดสอบ

| หัวข้อ | จุดประสงค์ |
|---|---|
| ระบบ Login / Role Management | ตรวจสอบว่าสิทธิ์ตามบทบาททำงานถูกต้อง |
| จัดการข้อมูลผู้ใช้ (เพิ่ม, Import Excel) | ตรวจสอบการนำเข้าข้อมูลชุดใหญ่จาก Excel |
| จัดการกุญแจ Real-time | Dashboard ต้องแสดงสถานะกุญแจอัปเดตทันทีผ่าน Socket |
| จัดการตารางสอน (Schedule / DailyAuth) | ระบบ Sync ตารางสอนต้องทำงานถูกต้องและอัตโนมัติ |
| ระบบโอน/สลับสิทธิ์ผ่านเว็บ | ตรวจสอบการโอน-สลับสิทธิ์จากฝั่ง Admin |

## 4.5 ส่วนที่ 4: การทดสอบความถูกต้องของข้อมูล (Data Testing)

ตรวจสอบว่าข้อมูลทุกประเภทถูกบันทึกลงฐานข้อมูลอย่างครบถ้วนและถูกต้อง ไม่มีข้อมูลขาดหายหรือผิดพลาด

## 4.6 ส่วนที่ 5: การทดสอบประสิทธิภาพ (Performance Testing)

**อ้างอิงขอบเขต 1.3.4.3 ก)** (ข้อกำหนดประสิทธิภาพของระบบ)

### 4.6.1 จุดประสงค์
วัดและยืนยันว่าระบบมีความแม่นยำ, ความเร็ว, และความเสถียรเพียงพอสำหรับการใช้งานจริง

### 4.6.2 หัวข้อการทดสอบ

| หัวข้อ | เกณฑ์ผ่าน | วิธีวัด |
|---|---|---|
| ความแม่นยำในการอ่าน NFC | ≥ 95% (ทดสอบ 50 ครั้ง) | นับจำนวนครั้งที่อ่านสำเร็จ |
| ความเร็วตอบสนอง NFC → ระบบ | ≤ 3 วินาที | จับเวลาตั้งแต่สแกนจนระบบแสดงผล |
| ความเร็วปลดล็อก Solenoid | ≤ 5 วินาที | จับเวลาตั้งแต่กดยืนยันจนล็อกเปิด |
| ความเร็วอัปเดตสถานะคืนกุญแจ | ≤ 5 วินาที | จับเวลาตั้งแต่เสียบกุญแจจนสถานะเปลี่ยน |
| ความเร็วโหลดหน้า Web Dashboard | ≤ 5 วินาที | วัดจาก Browser DevTools |
| ความเสถียร (Uptime) | ≥ 99% ใน 24 ชั่วโมง | ตรวจสอบ downtime ตาม log |
| Auto-reconnect หลัง Wi-Fi หลุด | ≥ 80% (ทดสอบ 5 ครั้ง) | ถอด-เสียบ Wi-Fi แล้ววัดว่าระบบ recover ได้ |

**จำนวน Test Cases**: 10 กรณีทดสอบ (PERF-01 ถึง PERF-10)

## 4.7 ส่วนที่ 6: การทดสอบฟีเจอร์เพิ่มเติม (Extended Features)

> ฟีเจอร์ที่พัฒนาเพิ่มเติมนอกเหนือจากขอบเขตเดิม เพื่อเพิ่มความน่าเชื่อถือและความสะดวกในการใช้งาน

### 4.7.1 จุดประสงค์
ทดสอบฟีเจอร์ที่ทีมพัฒนาออกแบบเพิ่มเติมเพื่อแก้ปัญหาที่พบในการใช้งานจริง

### 4.7.2 หัวข้อการทดสอบ

| ฟีเจอร์ | จุดประสงค์ | เหตุผลที่พัฒนาเพิ่ม |
|---|---|---|
| Auto-Reconciliation (Auto-Return) | ระบบตรวจจับและปิด Booking อัตโนมัติเมื่อเสียบกุญแจโดยไม่กดเมนู | แก้ปัญหา Booking ค้างเมื่อผู้ใช้ลืมกดคืน |
| Reconcile Scan ตอนกดเบิก | ตรวจสอบกุญแจทุกช่องก่อนแสดงรายการเบิก | ป้องกันแสดงห้องว่างที่ความจริงมีกุญแจอยู่แล้ว |
| ป้องกันการเสียบผิดช่อง | แจ้งเตือนทันทีเมื่อเสียบกุญแจผิดช่อง | ลดข้อผิดพลาดจากผู้ใช้ |
| สิทธิ์พิเศษครู/เจ้าหน้าที่ | ครูและเจ้าหน้าที่ข้ามการตรวจสอบ Schedule Conflict ได้ | รองรับการใช้งานฉุกเฉินและการสอนนอกตาราง |
| Docker Deployment | ระบบ Deploy ด้วย Docker Compose ได้อย่างสมบูรณ์ | เพิ่มความสะดวกในการ Deploy บน Server |

**จำนวน Test Cases**: 9 กรณีทดสอบ (EXT-01 ถึง EXT-09)

## 4.9 แนวทางการประเมินแต่ละส่วน

การประเมินดำเนินการใน 3 รูปแบบ:

1. **Black-box Testing**: ทดสอบจากมุมมองผู้ใช้งาน ไม่สนใจโค้ดภายใน วัดว่าผลลัพธ์ตรงตามที่คาดหวังหรือไม่
2. **Integration Testing**: ทดสอบการทำงานร่วมกันของส่วนประกอบต่างๆ เช่น NFC → Hardware → Backend → UI
3. **Performance Testing**: วัดประสิทธิภาพด้วยการทดสอบซ้ำหลายครั้งและนำผลมาเฉลี่ย

> ผลการประเมินจะนำเสนอในบทที่ 4 พร้อมตารางสรุปผลที่แท้จริงของแต่ละ Test Case
