**Slide 3: ที่มาและความสำคัญ (Background & Significance)**
*   **หัวข้อ:** 1.1 ที่มาและความสำคัญของปัญหา (Context & Significance)
*   **เนื้อหา:**
    *   **บริบทการใช้งาน:** อาคารเรียนรวมหรือห้องปฏิบัติการที่มีการใช้งานหนาแน่น (High-traffic Building) มีกุญแจ 50+ ดอก และผู้ใช้งาน 500+ คน หมุนเวียนเปลี่ยนมือทุก 1-2 ชั่วโมง
    *   **ปัญหาด้านความปลอดภัย (Security Gap):**
        *   การระบุตัวตน (Authentication) ในระบบเดิมทำได้ยาก (เช่น การฝากบัตร, การปลอมแปลงลายเซ็น)
        *   ความเสี่ยงกุญแจสูญหาย (Key Loss) โดยไม่สามารถระบุ "ผู้รับผิดชอบล่าสุด (Current Holder)" ได้ทันที
    *   **คอขวดด้านประสิทธิภาพ (Efficiency Bottleneck):**
        *   **The Return Loop:** ผู้ใช้ต้องเดินกลับมาที่ตู้เสมอเพื่อคืนกุญแจ แม้จะอยู่หน้าห้องเรียนแล้ว ทำให้เสียเวลาเดิน (Time Wastage)
        *   คอขวดช่วงเปลี่ยนคาบเรียน (Rush Hour) เกิดแถวยาวรอเซ็นชื่อที่เคาน์เตอร์เจ้าหน้าที่
    *   **ความถูกต้องของข้อมูล (Data Integrity):**
        *   ระบบสมุดบันทึก (Paper-based) มีความผิดพลาดสูง (Human Error) ข้อมูลสูญหาย หรืออ่านไม่ออก
        *   ข้อมูลไม่ใช่ข้อมูลปัจจุบัน (Stale Data) ไม่สามารถนำมาวิเคราะห์ (Analytics) ได้ทันท่วงที
*   **Speaker Notes:** "ในบทที่ 1 เราได้วิเคราะห์ถึงหัวใจของปัญหาครับ สภาพแวดล้อมอาคารที่มีคนใช้งานหลายร้อยคน แต่เรายังพึ่งพาระบบสมุดจด (Manual) ซึ่งมีช่องโหว่ทั้งเรื่องความปลอดภัย และความไร้ประสิทธิภาพอย่างมาก โดยเฉพาะ 'The Return Loop' ที่บังคับให้คนต้องเดินกลับไปกลับมาเพียงเพื่อแค่คืนกุญแจ ทั้งที่เขาสามารถส่งต่อให้เพื่อนคาบถัดไปได้ทันทีหากมีระบบที่ดีพอ นี่คือจุดเริ่มต้นที่ทำให้เราพัฒนา KMS ขึ้นมาเพื่อเปลี่ยนจากระบบที่พึ่งพา 'ความไว้ใจ' เป็นระบบที่พึ่งพา 'ข้อมูลที่ตรวจสอบได้จริง' ครับ"

**Slide 4: การวิเคราะห์ปัญหาที่ 1: ความปลอดภัย (Security Gap)**
*   **หัวข้อ:** ปัญหาที่ 1: ช่องโหว่ด้านความปลอดภัยและการระบุตัวตน
*   **เนื้อหา:**
    *   **Impersonation:** การฝากเพื่อนเบิกกุญแจโดยใช้บัตรคนอื่น
    *   **Unauthorized Access:** การเบิกกุญแจห้องอื่นที่ไม่มีสิทธิ์เข้า
    *   **Static Keys:** กุญแจแบบฟิสิกส์ไม่มีระบบบันทึกเวลาที่ตัวมันเอง
*   **Speaker Notes:** "ช่องว่างความปลอดภัยที่ใหญ่ที่สุดคือการระบุตัวตนครับ การใช้บัตรนักศึกษาหรือการเซ็นชื่อ มันคือสิ่งที่ 'อะไรก็ได้' มาอ้างครับ เราต้องการระบบที่ใช้ 'ตัวบุคคล' จริงๆ นั่นคือ Biometrics เพื่อปิดช่องว่างการสวมสิทธิ์เหล่านี้ครับ"

**Slide 5: ข้อจำกัดด้านเวลาและความคล่องตัว (Efficiency Bottleneck)**
*   **หัวข้อ:** ปัญหาที่ 2: ความไร้ประสิทธิภาพในการหมุนเวียน
*   **เนื้อหา:**
    *   **The Return Loop:** ผู้ใช้ต้องเดินกลับมาตู้เสมอเพื่อคืนกุญแจ แม้จะอยู่หน้าห้องเรียนแล้ว
    *   **Double Effort:** อาจารย์คาบถัดไปต้องนั่งรอกุญแจที่ตู้ หรือเสียเวลาเดินไปรับที่ห้องผู้สอนคนก่อน
    *   **Paperwork Overhead:** การจดบันทึกลงสมุดสร้างภาระงานขยะ และยากต่อการสืบค้นย้อนหลัง
    *   **Rush Hour Syndrome:** คอขวดที่ตู้กุญแจในช่วงเวลา 8:30 น. และ 16:30 น.
*   **Speaker Notes:** "เรื่องประสิทธิภาพก็น่าสนใจครับ เคยไหมครับที่อาจารย์คาบใหม่ต้องมานั่งรอกุญแจที่โต๊ะเจ้าหน้าที่ ทั้งๆ ที่กุญแจนั้นวางอยู่ที่ห้องเรียนแล้ว การที่ต้องเดินกลับไปกลับมาเพียงเพื่อ 'ทำตามระบบเดิม' คือสิ่งที่ KMS จะเข้ามาแก้ด้วยฟีเจอร์ Transfer ครับ"

**Slide 6: ความผิดพลาดของข้อมูล (Data Integrity Issues)**
*   **หัวข้อ:** ปัญหาที่ 3: ความไม่ถูกต้องของข้อมูลและการลงโทษ
*   **เนื้อหา:**
    *   **Human Error:** การจดเวลาผิด การลืมจดชื่อ หรือการอ่านลายมือไม่ออก
    *   **Stale Information:** เจ้าหน้าที่ไม่รู้ว่ากุญแจว่างหรือไม่ จนกว่าจะเดินไปเช็คที่ตู้จริง
    *   **Inconsistent Penalty:** การคิดค่าปรับหรือตัดคะแนนทำได้ยาก และไม่ยุติธรรมหากข้อมูลตกหล่น
    *   **Reporting:** การสรุปสถิติการใช้ห้องต่อเดือนใช้เวลาทำเป็นวันๆ บน Excel
*   **Speaker Notes:** "ข้อมูลที่ไม่เป็นปัจจุบัน (Stale Data) คืออุปสรรคของการบริหารอาคารครับ ระบบเดิมเราจะรู้ว่ากุญแจหายก็ต่อเมื่อถึงเวลาปิดอาคารแล้ว แต่ KMS จะให้เราเห็นสถานะกุญแจแต่ละดอกแบบวินาทีต่อวินาทีบน Dashboard กลางครับ"

**Slide 7: สรุปปัญหาที่เป็นหัวใจ (Core Pain Points Summary)**
*   **หัวข้อ:** สรุปหัวใจของปัญหา (Core Pain Points)
*   **เนื้อหา:**
    *   **Identity (ใคร?):** ต้องระบุตัวตนด้วย Face Recognition
    *   **Control (อย่างไร?):** ต้องควบคุมทางอิเล็กทรอนิกส์ (Solenoid)
    *   **Tracking (ที่ไหน? เมื่อไหร่?):** ต้องบันทึก Log ลง Database อัตโนมัติ
    *   **Flexibility (ทำอย่างไรให้คล่องตัว?):** ต้องรองรับการโอนสิทธิ์ (Hand-off)
*   **Speaker Notes:** "สรุปชัดๆ คือเราต้องการระบบที่ตอบโจทย์ 4 อย่างครับ: ใครใช้? (Face Scan), คุมยังไง? (Smart Lock), ตามได้ไหม? (Real-time DB), และต้องใช้งานได้คล่อง (Transfer Logic) ครับ"

**Slide 8: ตารางวิเคราะห์เปรียบเทียบระบบ (Comprehensive Comparison)**
*   **หัวข้อ:** เปรียบเทียบระบบเดิม vs ระบบ KMS
*   **เนื้อหา:** (ตารางเปรียบเทียบ)
    *   [มิติ | ระบบสมุดจด | ระบบ KMS อัจฉริยะ]
    *   ระบุตัวตน | บัตร/ชื่อ (แอบอ้างได้) | Face Scan (สวมสิทธิ์ไม่ได้)
    *   การล็อค | ใส่ลิ้นชัก/แขวน (หยิบผิดง่าย) | Electronic Lock (ปลดล็อกเฉพาะช่อง)
    *   การส่งต่อ | ต้องคืนที่เคาน์เตอร์เท่านั้น | โอนสิทธิ์ผ่านตู้ได้ทันที (Transfer)
    *   ความถูกต้อง | ตรวจยาก (Human Error) | NFC Tag Verification (100% ถูกต้อง)
    *   การตรวจสอบ | ต้องไล่เปิดสมุดรายวัน | ค้นหาด้วย Search Engine (Immediate)
*   **Speaker Notes:** "สไลด์นี้เปรียบเทียบให้เห็นชัดๆ ครับ ว่าเราไม่ได้แค่เปลี่ยนสมุดเป็นคอมพิวเตอร์ แต่เราเปลี่ยน 'โครงสร้างการทำงาน' ทั้งระบบให้เป็นแบบไร้รอยต่อ (Seamless Integration) ครับ"

**Slide 9: ฮาร์ดแวร์ที่เลือกใช้: ZKTEco SmartAC1**
*   **หัวข้อ:** ทำไมต้อง ZKTEco SmartAC1? (Hardware Precision)
*   **เนื้อหา:**
    *   **Visible Light Facial Recognition:** ตรวจพบใบหน้าได้ไกลถึง 2 เมตร
    *   **Proactive Face Identification:** ไม่ต้องหยุดเดินเพื่อสแกนหน้า
    *   **High Stability:** เป็นเกรด Enterprise ที่ทนความร้อนและการใช้งานหนัก
    *   **Integration Ready:** รองรับโปรโตคอลการสื่อสาร ADMS PUSH
*   **Speaker Notes:** "ในการเลือกฮาร์ดแวร์ เราเลือก ZKTEco SmartAC1 ครับ เพราะมันไม่ใช่แค่กล้อง แต่เป็นระบบ AI Facial Recognition ระดับอุตสาหกรรมที่อ่านใบหน้าได้เร็วมากแม้ผู้ใช้จะกำลังเดินอยู่ก็ตาม ช่วยลดคิวหน้าตู้กุญแจได้จริงครับ"

**Slide 10: การสื่อสารระดับโปร: ADMS Protocol**
*   **หัวข้อ:** สถาปัตยกรรม ADMS (PUSH Technology)
*   **เนื้อหา:**
    *   **Real-time Synchronization:** ส่งข้อมูลจากเครื่องสแกนไป Backend ทันที่ที่มีคนสแกน
    *   **Centralized Management:** คุมเครื่องสแกนหลายตัวได้จาก Server เดียว
    *   **Network Resilience:** หาก Internet หลุด เครื่องจะเก็บ Log ไว้และ PUSH ใหม่เมื่อต่อได้
    *   **HTTP/HTTPS Bridge:** สื่อสารกับ Node.js Backend ได้อย่างราบรื่น
*   **Speaker Notes:** "หัวใจของการเชื่อมต่อคือ ADMS ครับ มันคือเทคโนโลยี PUSH ที่ทำให้เครื่องสแกนใบหน้าคุยกับ Server หลังบ้านของเราได้ตลอดเวลาแบบ Real-time โดยไม่ต้องผ่านแอปพลิเคชันคั่นกลาง (Middleware) ให้ยุ่งยากครับ"

## ส่วนที่ 2: ระเบียบวิธีวิจัยและการวิเคราะห์ระบบ (Slides 11-23)

**Slide 11: ระเบียบวิธีวิจัยแบบ Agile (The Agile Mindset)**
*   **หัวข้อ:** ทำไมต้อง Agile สำหรับโครงการ KMS?
*   **เนื้อหา:**
    *   **Uncertainty:** ความไม่แน่นอนของพฤติกรรมการคืนกุญแจของนักศึกษาในสถานการณ์จริง
    *   **Continuous Feedback:** การทดสอบกับตู้จริงตั้งแต่วันแรกที่ประกอบเสร็จ
    *   **Flexibility:** การปรับเปลี่ยน Database Schema ได้ทันทีเมื่อพบ Case การโอนกุญแจที่ซับซ้อน
    *   **Incremental Delivery:** เริ่มจาก "เปิดกลอนได้" -> "สแกนใบหน้าได้" -> "โอนสิทธิ์ได้"
*   **Speaker Notes:** "เราเลือกใช้ Agile เพราะโปรเจกต์นี้มีตัวแปรเยอะมากครับ ทั้งพฤติกรรมคนและข้อจำกัดฮาร์ดแวร์ ถ้าเราใช้วิธีเดิมๆ เราไม่มีทางรู้เลยว่านักศึกษาจะชอบ 'สแกนก่อนเสียบ' หรือ 'เสียบก่อนสแกน' จนกว่าจะได้สร้างตัวต้นแบบออกมาให้เขาลองจริงๆ ครับ"

**Slide 12: วงจรการพัฒนา (Sprint Iterations)**
*   **หัวข้อ:** วงจรการทำงานแต่ละรอบ (Iterative Cycles)
*   **เนื้อหา:**
    *   **Sprint Duration:** รอบละ 2 สัปดาห์ เพื่อความกระชับ
    *   **Prototyping:** การสร้างวงจรทดสอบบน Breadboard ก่อนลงกล่องจริง
    *   **User Feedback Integration:** เช่น การเพิ่มเสียง Beep และไฟ LED หลังผู้ใช้บ่นว่าไม่รู้ว่าปลดล็อคหรือยัง
    *   **Pivot Point:** การตัดสินใจเปลี่ยนจาก RPi Face Scan มาเป็น ZKTEco เพื่อความเสถียรสูงสุด
*   **Speaker Notes:** "ในทุกๆ 2 สัปดาห์ เราจะมีของใหม่มาโชว์ครับ เช่น รอบแรกเราทำให้ไฟติด รอบสองเราทำให้สแกนใบหน้าผ่าน และรอบสำคัญคือตอนที่เราเอา Feedback จากเพื่อนๆ มาปรับปรุง เช่น 'ถ้าระบบไม่บอกว่าช่องไหนเปิด เพื่อนจะหาไม่เจอ' เราจึงเพิ่มระบบไฟ LED บอกตำแหน่งช่องกุญแจเข้าไปครับ"

**Slide 13: รายละเอียด Sprints และ Milestone สำคัญ**
*   **หัวข้อ:** บทสรุปการเดินทางของโปรเจกต์ (Milestones)
*   **เนื้อหา:**
    *   **Sprint 1 (Foundation):** ออกแบบ ER-Diagram และทำระบบ Check-in/Check-out พื้นฐาน
    *   **Sprint 2 (Hardware Core):** พัฒนาโปรโตคอล Serial JSON เพื่อคุยกับ ESP8266 และคุม Solenoid

*   **Speaker Notes:** "นี่คือ Roadmap ที่เราผ่านมาครับ เริ่มจากการวางรากฐานซอฟต์แวร์ ไปจนถึงการทำนวัตกรรมขั้นสูงอย่างระบบโอนกรรมสิทธิ์กุญแจ ซึ่งแต่ละช่วงเราได้เรียนรู้ปัญหาจริงและแก้ไขมันใน Sprint ถัดไปเสมอครับ"

**Slide 14: การวิเคราะห์ Use Case (System Interaction)**

*   **Speaker Notes:** "ภาพรวมของ Use Case ของเราครอบคลุมทุกมิติครับ ตั้งแต่เด็กที่เดินมาเบิกตามตารางสอน ไปจนถึงอาจารย์ที่ต้องการโอนสิทธิ์ด่วนให้เจ้าหน้าที่คนอื่น ทุกอย่างถูกออกแบบให้ทำงานร่วมกันบนฐานข้อมูลชุดเดียวครับ"

**Slide 15: Use Case: Student Scenario (Happy Path & Exceptions)**

**Slide 16: Use Case: Staff/Admin (Management Layer)**
*   **หัวข้อ:** รายละเอียดการใช้งาน: เจ้าหน้าที่และแอดมิน
*   **เนื้อหา:**
    *   **Remote Unlock:** ความสามารถในการปลดล็อกช่องกุญแจผ่านหน้าเว็บ (กรณีฉุกเฉิน)
    *   **Authorization Override:** การให้สิทธิ์เบิกกุญแจชั่วคราว (เช่น กรณีจัดกิจกรรมพิเศษ)
    *   **Blacklist Management:** การระงับสิทธิ์ชั่วคราวสำหรับผู้ที่มียอด Penalty สะสมเกินกำหนด
    *   **Audit Trail:** การดู Log ย้อนหลังว่า ใคร-ทำอะไร-ที่ตู้ไหน-เวลาเท่าไหร่ พร้อมรูปถ่าย (ถ้ามี)
*   **Speaker Notes:** "ในฝั่งเจ้าหน้าที่ เรามีอำนาจการควบคุมที่เหนือกว่าครับ เช่น ถ้ากุญแจติดขัด แอดมินสามารถสั่ง Unlock จากระยะไกลได้ทันที หรือถ้าใครทำผิดกฎซ้ำซาก ระบบจะมีระบบ Blacklist อัตโนมัติเพื่อระงับการเบิกจนกว่าจะได้รับการแก้ไขครับ"

**Slide 17: การจำลอง Process Flow (Workflow Analysis)**
*   **หัวข้อ:** ลำดับขั้นตอนการทำงาน (Workflow)
*   **เนื้อหา:**
    *   **ตรวจสอบสิทธิ์ (Auth Check):** ดึงข้อมูลจาก `DailyAuthorization` ตรวจสอบห้องและเวลา
    *   **ประมวลผลคำสั่ง (Command Processing):** Backend สร้าง JSON ส่งไปยัง ESP ผ่าน Socket.io
    *   **ยืนยันผลทางกายภาพ (Physical Feedback):** MFRC522 ในช่องกุญแจส่ง UID กลับมาเพื่อยืนยันว่ากุญแจออก/เข้าจริง
    *   **อัปเดตสถานะ (State Completion):** ปิด Booking และบันทึกเวลาที่ทำรายการจริงใน Database
*   **Speaker Notes:** "นี่คือสิ่งที่เกิดขึ้นเบื้องหลังครับ ระบบจะทำงานประสานกัน 3 ส่วน: ฐานข้อมูลเช็คสิทธิ์, Backend ส่งคำสั่ง และ Hardware ยืนยันผลกลับมา ถ้ากุญแจไม่ออกจากช่องจริงๆ ระบบจะไม่ถือว่าเบิกสำเร็จครับ"

**Slide 18: Activity Diagram: กตรรกะการเบิก (Borrowing Logic)**
*   **หัวข้อ:** แผนภาพแสดงการเบิกกุญแจ (Borrow Activity)
*   **เนื้อหา:**
    *   **Face Identification:** ZKTEco ส่ง UserID มาที่ Backend
    *   **Authorization Match:** ระบบหา Record `DailyAuthorization` ที่ `startTime <= currentTime <= endTime`
    *   **Key Availability:** เช็คว่ากุญแจนั้นไม่ได้ถูกใครเบิกไปก่อนหน้า (`status = AVAILABLE`)
    *   **Unlock Command:** ส่งสัญญาณไฟกระพริบที่ช่อง และปลดล็อค Solenoid
*   **Speaker Notes:** "ขั้นตอนการเบิกเป็นแบบ Zero-Trust ครับ คือถ้าไม่ถึงเวลาเรียน หรือไม่ใช่คนที่ได้รับมอบหมาย แม้จะสแกนใบหน้าผ่าน ระบบก็จะบอกว่า 'คุณไม่ได้รับอนุญาต' ทันทีครับ"

**Slide 19: Activity Diagram: กตรรกะการคืน (Returning Logic)**
*   **หัวข้อ:** แผนภาพแสดงการคืนกุญแจ (Return Activity)
*   **เนื้อหา:**
    *   **Self-Identification:** ผู้คืนสแกนใบหน้า (ป้องกันคนอื่นรอยแอบคืนแทน)
    *   **Status Prompt:** ระบบดึงรายการที่ผู้ใช้กำลังเบิกอยู่ขึ้นมาโชว์
    *   **NFC Verification:** เมื่อเสียบลงช่อง ระบบอ่านค่า UID ของ NFC Sticker ที่ตัวกุญแจ
    *   **Late Penalty:** คำนวณความต่างของเวลา ถ้าเกินกำหนด ระบบจะสร้าง Record ในตาราง `PenaltyLog`
*   **Speaker Notes:** "การคืนกุญแจเราเน้นที่ความรับผิดชอบครับ ระบบจะคำนวณ 'ค่าปรับ' หรือ 'คะแนนวินัย' ทันทีที่เสียบกุญแจคืน หากคุณคืนช้า ระบบจะแจ้งเตือนบนหน้าจอให้ทราบทันทีครับ"

**Slide 20: นวัตกรรม: ตรรกะการโอนสิทธิ์ (Direct Transfer Logic)**
*   **หัวข้อ:** นวัตกรรม: การโอนกุญแจแบบไม่ต้องคืนตู้
*   **เนื้อหา:**
    *   **The Concept:** คนที่ 1 (ถืออยู่) สแกนสิทธิ์ -> คนที่ 2 (รับต่อ) สแกนยืนยัน -> กด Confirm
    *   **Database Transaction:** 
        *   1. ปิด Booking เดิมของคนที่ 1 (Status: MOVED)
        *   2. สร้าง Booking ใหม่ให้คนที่ 2 (Status: BORROWED) ทันที
    *   **Efficiency Gain:** ลดการเดินไป-กลับ ระหว่างห้องเรียนและตู้กุญแจได้ 100% สำหรับผู้รับสิทธิ์ต่อ
*   **Speaker Notes:** "นี่คือไฮไลท์ครับ! ปกติคุณต้องเดินกลับไปตู้เพื่อคืนให้เพื่อนเบิกใหม่ แต่ KMS ให้คุณทั้งคู่มาสแกนหน้าคู่กันที่ตู้ แล้วกด 'โอน' กุญแจดอกนั้นในมือคุณจะกลายเป็นของเพื่อนในระบบทันที โดยไม่ต้องเอาไปเสียบเสียเวลาครับ"

**Slide 21: นวัตกรรม: ตรรกะการสลับห้อง (Key Swap Logic)**
*   **หัวข้อ:** นวัตกรรม: การสลับสิทธิ์การถือครองกุญแจ (Twin-Transfer)
*   **เนื้อหา:**
    *   **Problem:** นาย A ถือกุญแจห้อง 1, นาย B ถือกุญแจห้อง 2 แต่อยากสลับกันใช้
    *   **Complex Logic:** ต้องทำการ Update ข้อมูล 4 ส่วนพร้อมกัน (Atomic Update)
    *   **Integrity:** หากการอัปเดตข้อมูลของคนใดคนหนึ่งพลาด ระบบจะ Rollback ทั้งหมด (ไม่พังครึ่งเดียว)
    *   **Verification:** ต้องการลายพิมพ์ใบหน้าของทั้ง 2 ฝ่ายในการอนุมัติพร้อมกัน
*   **Speaker Notes:** "กรณีสลับห้อง เราทำได้ลึกลึ้งกว่าเดิมครับ คือการสลับสิทธิ์กุญแจ 2 ดอกในมือคน 2 คนพร้อมกัน ระบบจะทำการ 'แลกเปลี่ยนสิทธิ์' ในฐานข้อมูลให้ทันทีในทรานแซกชันเดียว เพื่อป้องกันข้อมูลผิดพลาดครับ"

**Slide 22: สถาปัตยกรรมข้อมูล (ER-Diagram & Schema)**
*   **หัวข้อ:** การออกแบบฐานข้อมูล (Database Schema)
*   **เนื้อหา:**
    *   **User Table:** เก็บข้อมูลใบหน้าและบทบาท
    *   **Room/Key Table:** เก็บ Metadata ของห้องและช่องที่ติดตั้ง
    *   **DailyAuthorization:** ตารางที่แปลง Schedule เป็นสิทธิ์รายวัน (หัวใจสำคัญ)
    *   **Booking Table:** เก็บ Transaction ทุกอย่าง (History of who, when, what)
    *   **Penalty Table:** บันทึกความผิดและคะแนนวินัย
*   **Speaker Notes:** "ฐานข้อมูลเราออกแบบตามมาตรฐานสากลครับ มีการทำความสัมพันธ์แบบ Relational ที่แข็งแรงมาก เพื่อให้เราสามารถดึงรายงานการใช้งานห้องมาวิเคราะห์ความหนาแน่นได้ในคลิกเดียวครับ"

**Slide 23: หัวใจของระบบ: DailyAuthorization Concept**
*   **หัวข้อ:** DailyAuthorization: สะพานระหว่างตารางสอนและสิทธิ์จริง
*   **เนื้อหา:**
    *   **Calculation:** ระบบจะดึงตารางสอนมา Generate สิทธิ์ล่วงหน้าทุกเที่ยงคืน
    *   **Override Support:** หากอาจารย์ยกเลิกคลาส แอดมินลบสิทธิ์รายวันทิ้ง นักศึกษาจะเบิกไม่ได้ทันที
    *   **Grace Period:** การยอมให้เบิกก่อนเวลาเรียน [X] นาที และคืนเลทได้ [Y] นาที
    *   **Optimization:** ลดภาระการ Query จากตาราง Schedule ขนาดใหญ่ในเวลา Runtime
*   **Speaker Notes:** "DailyAuthorization คือฟิลเตอร์ชั้นดีครับ แทนที่จะไปไล่ดูตารางเรียนทั้งหมด เราแค่ดูว่า 'วันนี้ เวลานี้ ใครมีสิทธิ์บ้าง' ทำให้ระบบทำงานเร็วมากแม้ข้อมูลจะมหาศาลครับ"

## ส่วนที่ 3: การออกแบบทางเทคนิคและสถาปัตยกรรม (Slides 24-34)

**Slide 24: ภาพรวมสถาปัตยกรรม 5 เลเยอร์ (The 5-Layered Architecture)**
*   **หัวข้อ:** โครงสร้างระบบแบบแยกส่วน (Decoupled Design)
*   **เนื้อหา:**
    *   **Layer 1: User Interface (React):** แสดงผลแบบ Real-time (Single Page App)
    *   **Layer 2: API Gateway (Express):** ควบคุมสิทธิ์และจัดการ Request
    *   **Layer 3: Communication Hub (Socket.io):** เชื่อมโยง Software และ Hardware เข้าด้วยกัน
    *   **Layer 4: Persistence (PostgreSQL):** เก็บข้อมูลที่มีความสัมพันธ์ซับซ้อนอย่างมั่นคง
    *   **Layer 5: Physical Hardware (ZKTEco & ESPNode):** ด่านหน้าการระบุตัวตนและปลดล็อก
*   **Speaker Notes:** "ระบบเราไม่ได้สร้างแบบก้อนเดียวจบครับ แต่เราใช้สถาปัตยกรรมแบบ 5 เลเยอร์ เพื่อให้แต่ละส่วนทำงานอิสระต่อกัน ถ้า Hardware ตัวหนึ่งเสีย ระบบซอฟต์แวร์ชั้นบนยังทำงานต่อได้ และเรายังสามารถเปลี่ยนเครื่องสแกนใบหน้ายี่ห้ออื่นมาเสียบแทนได้ในอนาคตโดยไม่ต้องแก้โค้ดทั้งหมดครับ"

**Slide 25: ส่วนแสดงผลอัจฉริยะ (Frontend Engineering - React)**
*   **หัวข้อ:** การออกแบบ Frontend สำหรับตู้ Kiosk
*   **เนื้อหา:**
    *   **Vite Integration:** เพื่อการรวบรวมไฟล์ที่เร็วที่สุด (Zero Latency)
    *   **State Management:** ใช้ Context API จัดการสถานะการล็อกอินของผู้ใช้จากเครื่องสแกนหน้า
    *   **Responsive UI:** ปรับขนาดปุ่มให้เหมาะกับหน้าจอ Touchscreen ของตู้
    *   **Auto-Reset:** ระบบลบข้อมูลการล็อกอินอัตโนมัติหากไม่มีการทำรายการภายใน 30 วินาที
*   **Speaker Notes:** "ในส่วนของ Frontend เราเน้นที่ความเร็วและความง่ายครับ นอกจากจะสวยงามแล้ว เรายังมีระบบ Auto-Reset เพื่อความปลอดภัย ถ้าผู้ใช้สแกนหน้าทิ้งไว้แล้วเดินหนีไป ระบบจะล้างข้อมูลสิทธิ์ทันทีเพื่อป้องกันคนอื่นมาแอบใช้ห้องแทนครับ"

**Slide 26: ระบบหลังบ้านประสิทธิภาพสูง (Backend Robustness - Node.js)**
*   **หัวข้อ:** โครงสร้าง Backend และความปลอดภัย API
*   **เนื้อหา:**
    *   **Event-Driven:** ใช้ Node.js เพื่อรองรับการทำงานแบบ I/O พร้อมๆ กัน (High Concurrency)
    *   **Middleware Security:** ระบบตรวจสอบ Token ทุกครั้งที่มีการสั่ง Unlock จากหน้าจอ
    *   **Log Engine:** ระบบบันทึกทุก Error และ Activity ของ Hardware ลงไฟล์เพื่อการ Debug
    *   **RESTful + Sockets:** ใช้ REST สำหรับข้อมูล Static และ Sockets สำหรับคำสั่ง Real-time
*   **Speaker Notes:** "หลังบ้านเราใช้ Node.js ครับ ซึ่งเก่งมากเรื่องการจัดการคิวงานพร้อมกันเยอะๆ เช่น ตอนเช้าที่มีคนมาคืนกุญแจพร้อมกัน 10 คน ระบบของเราจะยังลื่นไหลและไม่เกิดอาการค้างแน่นอนครับ"

**Slide 27: การสื่อสารไร้รอยต่อ (Socket.io Real-time Hub)**
*   **หัวข้อ:** การเชื่อม Hardware และ Software ด้วย Socket.io
*   **เนื้อหา:**
    *   **Bi-directional:** การส่งข้อมูลไป-กลับแบบสองทางระหว่าง Web และ Hardware
    *   **Namespace & Rooms:** แยกตู้กุญแจแต่ละตู้ด้วย ID เพื่อไม่ให้ส่งคำสั่งผิดตู้
    *   **Heartbeat Monitor:** ระบบเช็คสถานะ Online/Offline ของตู้กุญแจตลอดเวลา
    *   **Latency:** ความเร็วในการส่งคำสั่งต่ำกว่า 100ms
*   **Speaker Notes:** "Socket.io คือหัวใจของการเชื่อมต่อครับ ทันทีที่คุณกดยืนยันที่หน้าจอ คำสั่งปลดล็อกจะวิ่งไปยัง Hardware ภายในเสี้ยววินาที และถ้าตู้กุญแจเกิดหลุดการเชื่อมต่อ แอดมินจะได้รับการแจ้งเตือนทันทีผ่านระบบ Dashboard ครับ"

**Slide 28: ความปลอดภัยข้อมูลขั้นสูง (Prisma & Database Transactions)**
*   **หัวข้อ:** การจัดการ Transaction และความถูกต้องของข้อมูล
*   **เนื้อหา:**
    *   **Atomic Updates:** การใช้ `prisma.$transaction` เพื่อทำรายการแบบ "สำเร็จทั้งหมด หรือไม่สำเร็จเลย"
    *   **Data Integrity:** ป้องกันกรณี 'ของในมือโอนไปหาเพื่อนแล้ว แต่ในฐานข้อมูลยังเป็นชื่อเราอยู่'
    *   **Constraint Checks:** ระบบไม่ยอมให้มีการเบิกกุญแจห้องเดียวกันซ้ำซ้อน
    *   **SQL Injection Prevention:** ใช้ ORM เป็นกำแพงกั้นระหว่าง Code และ Database
*   **Speaker Notes:** "เราให้ความสำคัญกับความถูกต้องของข้อมูล (Data Integrity) สูงมากครับ โดยเฉพาะในฟีเจอร์การโอนกุญแจ เราใช้สิ่งที่เรียกว่า Transaction เพื่อให้มั่นใจว่าข้อมูลของผู้โอนและผู้รับจะถูกอัปเดตพร้อมกันแบบเป๊ะๆ ถ้ามีขั้นตอนใดผิดพลาด ระบบจะดึงข้อมูลกลับสถานะเดิมทันทีครับ"

**Slide 29: ส่วนระบุตัวตนอุตสาหกรรม (ZKTEco & ADMS Protocol)**
*   **หัวข้อ:** การประยุกต์ใช้ ADMS สำหรับการระบุตัวตน
*   **เนื้อหา:**
    *   **ADMS (Automatic Data Management System):** โปรโตคอลมาตรฐานโลก
    *   **PUSH Technology:** เครื่อง ZKTEco จะทำหน้าที่เป็น Client ส่งข้อมูลมาหา Server เอง
    *   **HTTP POST Flow:** เมื่อสแกนผ่าน เครื่องจะส่ง `UID` และ `Timestamp` มาที่ Backend
    *   **Seamless Integration:** ระบบ KMS จะรับ `UID` นั้นไป Mapping กับ `User` ในฐานข้อมูลทันที
*   **Speaker Notes:** "นี่คือส่วนที่เป็นมาตรฐานสากลครับ เราใช้โปรโตคอล ADMS ซึ่งเป็นระบบ PUSH Technology แทนที่ Server จะต้องไปถามเครื่องตลอดเวลา เครื่อง ZKTEco จะเป็นฝ่ายรายงานผลการสแกนใบหน้ามาหาเราเอง ทำให้ระบบปลอดภัยและลดภาระของ Network ได้มหาศาลครับ"

**Slide 30: สถาปัตยกรรม Hardware กระจายตัว (Distributed ESP Nodes)**
*   **หัวข้อ:** การออกแบบ Hardware แบบ Modular และ Scalability
*   **เนื้อหา:**
    *   **Main Gateway (Kiosk):** หน้าจอ Touchscreen และ Central Controller
    *   **ESP8266 Sub-Nodes:** คุมกลอนกุญแจแยกเป็นชุดๆ (เช่น 1 บอร์ดต่อ 4 ช่อง)
    *   **Serial Bus Connection:** เชื่อมต่อ Sub-Nodes หลายตัวเข้าหา Gateway ด้วยสายเส้นเดียว
    *   **Easy Maintenance:** หากช่องกุญแจเสีย 1 ช่อง สามารถถอดเปลี่ยนเฉพาะ Sub-Node นั้นได้
*   **Speaker Notes:** "Hardware ของเราถูกออกแบบแบบ Modular ครับ เหมือนกับตัวต่อเลโก้ ถ้าห้องเรียนในอนาคตมีกุญแจเพิ่มขึ้น เราแค่เสียบ Node ESP เพิ่มเข้าไป ระบบจะเจอช่องใหม่โดยอัตโนมัติโดยไม่ต้องเดินสายใหม่ทั้งหมดครับ"

**Slide 31: ยืนยันตัวตน 2 ชั้น (Dual-Factor Authentication Deep-dive)**
*   **หัวข้อ:** การยืนยันตัวตนระดับ Professional
*   **เนื้อหา:**
    *   **Factor 1 (Biometrics):** จดจำใบหน้าผ่าน SmartAC1 (ยืนยัน "ใคร")
    *   **Factor 2 (NFC Identity):** ตรวจสอบรหัส Unique UID จาก Sticker NFC (ยืนยัน "กุญแจดอกไหน")
    *   **Cross-verification:** การตรวจสอบว่า User ที่สแกนหน้า มีสิทธิ์ถือกุญแจ UID นี้จริงๆ หรือไม่
    *   **Anti-Fraud:** ป้องกันกรณีการขโมยกุญแจมาคืนแทนเพื่อน หรือการหยิบกุญแจผิดช่อง
*   **Speaker Notes:** "เราไม่ได้เชื่อแค่ใบหน้าอย่างเดียวครับ แต่ตัวกุญแจเองก็ต้องมี 'บัตรประชาชน' ของมัน นั่นคือ NFC Tag ครับ กระบวนการตรวจสอบแบบ 2 ปัจจัยนี้ทำให้มั่นใจได้ 100% ว่าคนที่คืนกุญแจ คือคนที่ถือความรับผิดชอบกุญแจดอกนั้นจริงๆ ครับ"

**Slide 32: วงจรควบคุมไฟฟ้าและระบบความปลอดภัย (Electronic Design)**
*   **หัวข้อ:** การออกแบบวงจรควบคุมไฟฟ้า (Circuit Design)
*   **เนื้อหา:**
    *   **Isolated Relay:** แยกวงจรไฟฟ้า 220V/12V ออกจากส่วนประมวลผล 5V ชัดเจน
    *   **Flyback Diode:** ป้องกันแรงดันไฟฟ้ากระชาก (Back EMF) จาก Solenoid ไม่ให้บอร์ดพัง
    *   **Power Supply:** ชุดจ่ายไฟแยกสำหรับ Solenoids เพื่อป้องกันระบบรีสตาร์ทเองเมื่อปลดล็อก
    *   **Slot Appearance:** มีไฟ LED แสดงสถานะ 'เขียว-เปิดได้ / แดง-ล็อคอยู่'
*   **Speaker Notes:** "ในเชิงไฟฟ้า เราใส่ใจเรื่องความปลอดภัยของอุปกรณ์มากครับ มีระบบป้องกันไฟกระชากและแยกชุดจ่ายไฟออกจากกัน เพื่อให้มั่นใจว่าตู้กุญแจจะทำงานต่อเนื่องได้ตลอด 24 ชั่วโมงโดยไม่มีอาการค้างหรือบอร์ดไหม้ครับ"

**Slide 33: โปรโตคอลการสื่อสาร Serial JSON (Software-Hardware Bridge)**
*   **หัวข้อ:** รูปแบบการสื่อสารข้อมูลชุดคำสั่ง (Data Protocol)
*   **เนื้อหา:**
    *   **Consistency:** ใช้ JSON Format ทั้งใน Software และ Hardware
    *   **Sample Command:** `{"type": "UNLOCK", "slot": 5}`
    *   **Sample Acknowledge:** `{"status": "SUCCESS", "nfc_uid": "A1B2C3D4"}`
    *   **Error Handling:** ระบบจะลองส่งคำสั่งใหม่ (Retries) 3 ครั้งหากไม่ได้รับ Acknowledgement
*   **Speaker Notes:** "สะพานเชื่อมระหว่างโค้ดเว็บและบอร์ดคุมคือโปรโตคอล JSON ครับ ซึ่งมนุษย์อ่านเข้าใจง่ายและพัฒนาง่าย เป็นมาตรฐานเดียวกับที่ใช้ออกแบบซอฟต์แวร์ระดับโลก ทำให้เราการตรวจสอบข้อผิดพลาด (Debug) ทำได้รวดเร็วมากครับ"

**Slide 34: ความปลอดภัยและความเป็นส่วนตัว (Security & Privacy Infrastructure)**
*   **หัวข้อ:** มาตรฐานความปลอดภัยของข้อมูล (Data Security)
*   **เนื้อหา:**
    *   **Encryption:** ข้อมูลสำคัญใน Database ถูกเข้ารหัสป้องกันการเข้าถึงจากภายนอก
    *   **Audit Trail:** ระบบบันทึกทุกเหตุการณ์ละเอียดถึงระดับวินาที (Forensic Ready)
    *   **Data Privacy:** ภาพถ่ายใบหน้าถูกเก็บและประมวลผลภายในเครื่อง ZKTEco ไม่รั่วไหลไปคลาวด์สาธารณะ
    *   **Session Management:** กำหนดอายุการขอสิทธิ์เบิกกุญแจให้สั้นที่สุด (Least Privilege)
*   **Speaker Notes:** "สุดท้ายคือเรื่องความปลอดภัยข้อมูลครับ เราบันทึก Log ทุกอย่างแบบ Audit Trail เพื่อให้สามารถตรวจสอบย้อนหลังได้ในศาลหากเกิดเหตุไม่คาดฝัน และเราให้ความสำคัญกับความเป็นส่วนตัวโดยไม่มีการเอาข้อมูลใบหน้านักศึกษาไปไว้นอกระบบของเราครับ"

**Slide 38: กรณีศึกษา: การโอนสิทธิ์หน้างาน (Case Study: Field Transfer)**

*   **Speaker Notes:** "นี่คือ Best Case เลยครับ ระบบเดิมเราต้องเสียเวลาเดินไปมาถึง 10 นาทีเพื่อแค่คืนกุญแจ แต่ด้วยการโอนสิทธิ์ของ KMS เราประหยัดเวลาส่วนนี้ไปได้ถึง 90% อาจารย์ท่านถัดไปสามารถสอนต่อได้ทันทีครับ"

    *   **Academic Integrity:** เสริมสร้างวินัยให้นักศึกษาผ่านระบบคะแนนและบทลงโทษอัตโนมัติ

*   **Speaker Notes:** "หากมองในแง่ความคุ้มค่า เราไม่ได้แค่ได้ตู้กุญแจครับ แต่เราได้ระบบรักษาความปลอดภัยระดับองค์กร และที่สำคัญคือเราได้สร้าง 'วินัยดิจิทัล' ให้กับนักศึกษา ซึ่งเป็น Soft Skill ที่สำคัญมากในยุคปัจจุบันครับ"
