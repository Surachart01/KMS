# ภาคผนวก ข: คู่มือการบำรุงรักษาระบบจัดการกุญแจ

## 1. บทนำ

คู่มือการบำรุงรักษาฉบับนี้จัดทำขึ้นเพื่อให้ผู้ดูแลระบบและเจ้าหน้าที่ฝ่ายเทคนิคสามารถดำเนินการบำรุงรักษาระบบจัดการกุญแจ ซึ่งต่อไปในคู่มือนี้จะเรียกว่า KMS อย่างถูกต้องและมีประสิทธิภาพ การบำรุงรักษาที่เหมาะสมและสม่ำเสมอจะช่วยให้ระบบทำงานได้อย่างต่อเนื่อง ลดโอกาสเกิดความขัดข้อง และยืดอายุการใช้งานของทั้งซอฟต์แวร์และฮาร์ดแวร์

วัตถุประสงค์หลักของคู่มือนี้มีดังนี้

ประการแรก เพื่อให้ผู้ดูแลระบบมีแนวทางการบำรุงรักษาที่ชัดเจนและปฏิบัติได้จริง ทั้งในส่วนของฮาร์ดแวร์และซอฟต์แวร์

ประการที่สอง เพื่อให้มีขั้นตอนการแก้ไขปัญหาที่เป็นระบบ ลดเวลาที่ระบบหยุดทำงานอันเนื่องมาจากความผิดพลาดต่าง ๆ

ประการที่สาม เพื่อป้องกันการสูญหายของข้อมูลผ่านกระบวนการสำรองข้อมูลที่มีกำหนดการชัดเจน

ประการที่สี่ เพื่อรักษาความปลอดภัยของระบบในระดับที่เหมาะสม

### 1.1 ขอบเขตของคู่มือ

คู่มือการบำรุงรักษาฉบับนี้ครอบคลุมระบบย่อยทั้งหมด 6 ส่วน ดังนี้

ส่วนที่ 1 ฮาร์ดแวร์ตู้กุญแจและอุปกรณ์ควบคุม ได้แก่ ตู้กุญแจ วงจรรีเลย์ Raspberry Pi และสายเชื่อมต่อต่าง ๆ

ส่วนที่ 2 เครื่องสแกนใบหน้า ZKTeco SmartAC1 รวมถึงฐานข้อมูลใบหน้าและการเชื่อมต่อโปรโตคอล ADMS

ส่วนที่ 3 ซอฟต์แวร์ Backend ซึ่งพัฒนาด้วย Node.js Express Prisma และ Socket.IO รันบน Docker Container

ส่วนที่ 4 ซอฟต์แวร์ Frontend ซึ่งพัฒนาด้วย Next.js รันบน Docker Container

ส่วนที่ 5 ระบบฐานข้อมูล PostgreSQL 15 และ Redis ที่รันบน Docker Container

ส่วนที่ 6 เครือข่ายและความปลอดภัยของระบบ

### 1.2 สถาปัตยกรรมระบบโดยรวม

ระบบ KMS ประกอบด้วยส่วนประกอบหลักที่เชื่อมต่อกันดังนี้

เครื่องแม่ข่ายหลักรัน Docker Compose ซึ่งบริหาร Container 4 ตัว ได้แก่ postgres รับผิดชอบฐานข้อมูลหลักบน port 34669 และ redis รับผิดชอบ Cache และ Session บน port 63797 และ backend ซึ่งเป็น API Server บน port 4556 และ frontend ซึ่งเป็น Web Dashboard บน port 80

Raspberry Pi ติดตั้งที่ตู้กุญแจ ทำหน้าที่รันซอฟต์แวร์ HardwareUI และเชื่อมต่อกับ Backend ผ่าน Socket.IO เพื่อรับคำสั่งควบคุมรีเลย์

ZKTeco SmartAC1 เป็นเครื่องสแกนใบหน้า เชื่อมต่อกับ Backend ผ่านโปรโตคอล ADMS เพื่อส่งผลการสแกนใบหน้า

### 1.3 ข้อกำหนดสำหรับผู้ดูแลระบบ

ผู้ที่รับผิดชอบการบำรุงรักษาระบบควรมีความรู้และทักษะดังต่อไปนี้

ความรู้พื้นฐานด้านระบบปฏิบัติการ Linux หรือ Debian เนื่องจาก Raspberry Pi รัน Raspberry Pi OS ซึ่งพัฒนาบาน Debian

ความรู้พื้นฐานเกี่ยวกับ Docker และ Docker Compose สำหรับการจัดการ Container

ความรู้เบื้องต้นเกี่ยวกับ PostgreSQL สำหรับการตรวจสอบและสำรองฐานข้อมูล

ทักษะการแก้ไขปัญหาเครือข่ายพื้นฐาน เช่น การตรวจสอบ IP และ Port

## 2. ฮาร์ดแวร์: ตู้กุญแจและ Raspberry Pi

### 2.1 ภาพรวมฮาร์ดแวร์ตู้กุญแจ

ตู้กุญแจประกอบด้วยส่วนประกอบหลักดังต่อไปนี้

ส่วนที่ 1 ช่องสล็อตกุญแจ แต่ละช่องมีสลักแม่เหล็กไฟฟ้าที่ควบคุมโดยวงจรรีเลย์ และมีชิปอ่าน NFC ติดตั้งอยู่ภายในเพื่อตรวจสอบการคืนกุญแจ

ส่วนที่ 2 บอร์ดรีเลย์ เป็นวงจรอิเล็กทรอนิกส์ที่ได้รับสัญญาณ GPIO จาก Raspberry Pi แล้วแปลงเป็นสัญญาณไฟฟ้าเพื่อควบคุมสลักแม่เหล็ก

ส่วนที่ 3 Raspberry Pi เป็นคอมพิวเตอร์ขนาดเล็กที่ทำหน้าที่เป็นตัวควบคุมหลัก รับคำสั่งจาก Backend Server ผ่าน Socket.IO และส่งสัญญาณ GPIO ไปยังบอร์ดรีเลย์

ส่วนที่ 4 แหล่งจ่ายไฟ ประกอบด้วยอะแดปเตอร์แปลงไฟ 220V AC เป็น 5V DC สำหรับ Raspberry Pi และ 12V DC สำหรับวงจรสลักแม่เหล็ก

ส่วนที่ 5 หน้าจอแสดงผลและกล้อง ZKTeco ติดตั้งอยู่ด้านนอกตู้ เป็นส่วนติดต่อกับผู้ใช้งาน

### 2.2 ตารางการบำรุงรักษาฮาร์ดแวร์ประจำสัปดาห์

เจ้าหน้าที่ควรตรวจสอบฮาร์ดแวร์ทุกสัปดาห์ตามรายการดังนี้

รายการที่ 1 ตรวจสอบความผิดปกติที่มองเห็นได้ภายนอกตู้กุญแจ ได้แก่ รอยขูดขีด รอยกระแทก ความเสียหายของสายเชื่อมต่อที่อยู่ภายนอก และการอัดอากาศ หรือฝุ่นสะสมที่ช่องระบายอากาศ

รายการที่ 2 ตรวจสอบไฟแสดงสถานะบน Raspberry Pi ไฟสีแดงต้องติดอยู่ตลอดซึ่งแสดงว่ามีไฟฟ้าจ่ายอยู่ ไฟสีเขียวจะกะพริบเป็นระยะซึ่งแสดงว่ามีการอ่านเขียนข้อมูล หากไฟสีแดงดับแสดงว่าไม่มีไฟจ่าย ให้ตรวจสอบอะแดปเตอร์และสายไฟ

รายการที่ 3 ทดสอบการทำงานของสล็อตกุญแจทุกช่อง โดยใช้หน้า Dashboard ของ Kiosk ทดสอบเบิกกุญแจทุกดอกเพื่อตรวจสอบว่าสลักแม่เหล็กทำงานได้ทุกช่อง บันทึกช่องที่ไม่ตอบสนองเพื่อนำไปซ่อมบำรุง

รายการที่ 4 ตรวจสอบการทำงานของชิป NFC โดยนำพวงกุญแจแต่ละดอกเสียบเข้าสล็อตแล้วรอดูว่าระบบตรวจจับการคืนได้หรือไม่ หากไม่ตรวจจับอาจเกิดจากชิป NFC ในสล็อตชำรุด ให้บันทึกหมายเลขสล็อตนั้น

รายการที่ 5 ตรวจสอบอุณหภูมิของ Raspberry Pi โดยใช้คำสั่ง vcgencmd measure_temp ผ่าน SSH อุณหภูมิปกติควรอยู่ต่ำกว่า 70 องศาเซลเซียส หากสูงกว่านั้นให้ตรวจสอบการระบายอากาศ

### 2.3 การเชื่อมต่อผ่าน SSH เพื่อตรวจสอบ Raspberry Pi

เจ้าหน้าที่ฝ่ายเทคนิคสามารถเชื่อมต่อกับ Raspberry Pi ผ่าน SSH เพื่อตรวจสอบและแก้ไขปัญหาได้โดยไม่ต้องอยู่ที่หน้าตู้กุญแจ

ขั้นที่ 1 ตรวจสอบให้แน่ใจว่าคอมพิวเตอร์ที่ใช้อยู่ในเครือข่ายเดียวกับ Raspberry Pi

ขั้นที่ 2 เปิด Terminal แล้วพิมพ์คำสั่ง ssh pi@IP_ADDRESS โดยแทน IP_ADDRESS ด้วย IP ของ Raspberry Pi ของตู้ที่ต้องการ

ขั้นที่ 3 ใส่รหัสผ่านเมื่อระบบร้องขอ รหัสผ่านเริ่มต้นคือ raspberry แต่ควรเปลี่ยนเป็นรหัสผ่านที่ปลอดภัยกว่าหลังตั้งค่าระบบ

ขั้นที่ 4 เมื่อเชื่อมต่อสำเร็จ สามารถใช้คำสั่งต่าง ๆ ในการตรวจสอบระบบได้ทันที

คำสั่งที่ใช้บ่อยในการตรวจสอบ Raspberry Pi มีดังนี้

คำสั่ง uptime ใช้ตรวจสอบว่าเครื่องทำงานมาแล้วกี่วันโดยไม่ได้รีสตาร์ท

คำสั่ง df -h ใช้ตรวจสอบพื้นที่เก็บข้อมูลบน SD Card คอลัมน์ Use% ควรต่ำกว่า 80 เปอร์เซ็นต์

คำสั่ง free -m ใช้ตรวจสอบการใช้ RAM คอลัมน์ available ควรมีค่ามากกว่า 100 MB เสมอ

คำสั่ง vcgencmd measure_temp ใช้ตรวจสอบอุณหภูมิของ CPU

คำสั่ง systemctl status kms-hardware ใช้ตรวจสอบสถานะของ Service HardwareUI ว่าทำงานอยู่หรือไม่

### 2.4 การทำความสะอาดฮาร์ดแวร์ประจำเดือน

ทุก 1 เดือน เจ้าหน้าที่ควรทำความสะอาดฮาร์ดแวร์ตามขั้นตอนดังนี้

ขั้นที่ 1 ปิดระบบให้ถูกต้องก่อนทำความสะอาด โดยส่งคำสั่งหยุดการทำงานผ่าน Web Dashboard หรือ SSH เสียก่อน ห้ามถอดปลั๊กโดยตรงโดยไม่ Shutdown ก่อน เนื่องจากอาจทำให้ SD Card ของ Raspberry Pi เสียหาย

ขั้นที่ 2 ใช้ลมอัดทำความสะอาดฝุ่นภายในตู้ โดยเฉพาะบริเวณรูระบายอากาศ และรอบ ๆ Raspberry Pi

ขั้นที่ 3 ใช้ผ้าแห้งเช็ดทำความสะอาดหน้าจอ ZKTeco และกล้องสแกนใบหน้า ห้ามใช้น้ำยาทำความสะอาดที่มีส่วนผสมของแอลกอฮอล์สูงบริเวณกล้อง

ขั้นที่ 4 ตรวจสอบหัวต่อสายทุกเส้นว่ายังเสียบแน่นดีหรือไม่ โดยเฉพาะสายที่เชื่อมต่อระหว่าง Raspberry Pi กับบอร์ดรีเลย์

ขั้นที่ 5 เปิดระบบกลับมาและทดสอบการทำงานทุกสล็อต

### 2.5 การเปลี่ยนอุปกรณ์ที่ชำรุด

การเปลี่ยน SD Card ของ Raspberry Pi เป็นขั้นตอนที่สำคัญและต้องดำเนินการอย่างระมัดระวัง เนื่องจาก SD Card เป็นอุปกรณ์ที่มีอายุการใช้งานจำกัด ขึ้นอยู่กับจำนวนการเขียนข้อมูล

ขั้นที่ 1 สำรองข้อมูลบน SD Card ก่อนเสมอโดยใช้คำสั่ง dd เพื่อ Clone ข้อมูลจาก SD Card เก่าไปยัง SD Card ใหม่

ขั้นที่ 2 ปิดเครื่อง Raspberry Pi ด้วยคำสั่ง sudo shutdown -h now แล้วรอให้ไฟสีเขียวดับก่อนถอดสาย

ขั้นที่ 3 ถอด SD Card เก่าออกและใส่ SD Card ใหม่ที่ Clone ข้อมูลมาแล้ว

ขั้นที่ 4 เปิดเครื่องและตรวจสอบว่าระบบ Boot ขึ้นมาได้ตามปกติ

การเปลี่ยนบอร์ดรีเลย์ ในกรณีที่รีเลย์บางช่องไม่ทำงาน ให้ตรวจสอบก่อนว่าสัญญาณ GPIO จาก Raspberry Pi ถึงบอร์ดรีเลย์หรือไม่ โดยใช้ Multimeter วัดแรงดันที่ขา Input ของรีเลย์ หากมีสัญญาณจาก GPIO แต่รีเลย์ไม่ทำงาน แสดงว่าตัวรีเลย์ชำรุดและต้องเปลี่ยนใหม่

### 2.6 ตารางบำรุงรักษาฮาร์ดแวร์ประจำปี

ทุก 6 เดือน ควรดำเนินการ ดังนี้

รายการที่ 1 ถอดและทำความสะอาดหัวต่อ GPIO ทั้งหมดระหว่าง Raspberry Pi และบอร์ดรีเลย์ด้วย Isopropyl Alcohol เพื่อป้องกันการเกิดออกไซด์

รายการที่ 2 ตรวจสอบอายุการใช้งานและสถานะของ SD Card โดยใช้คำสั่ง badblocks -v เพื่อตรวจหา Bad Sector

รายการที่ 3 อัปเดต Firmware ของ Raspberry Pi และ Raspberry Pi OS

รายการที่ 4 เปลี่ยนแบตเตอรี่ RTC หากมีการติดตั้ง เพื่อให้ระบบนาฬิกายังคงทำงานได้เมื่อไฟฟ้าดับ

ทุก 1 ปี ควรพิจารณาเปลี่ยน SD Card ใหม่โดยไม่รอให้ชำรุดก่อน เนื่องจาก SD Card ที่ใช้กับ Raspberry Pi แบบต่อเนื่องมีอายุการใช้งานจำกัด

## 3. ฮาร์ดแวร์: เครื่องสแกนใบหน้า ZKTeco SmartAC1

### 3.1 ภาพรวมอุปกรณ์ ZKTeco SmartAC1

ZKTeco SmartAC1 เป็นเครื่องสแกนใบหน้าที่ใช้เทคโนโลยี Deep Learning ในการจดจำใบหน้า รองรับการจัดเก็บข้อมูลใบหน้าได้สูงสุดตามที่กำหนดในรุ่น อุปกรณ์เชื่อมต่อกับ Backend Server ผ่านโปรโตคอล ADMS ซึ่งเป็นโปรโตคอลเฉพาะของ ZKTeco สำหรับการสื่อสารระหว่างอุปกรณ์และเซิร์ฟเวอร์

อุปกรณ์มีพอร์ตเชื่อมต่อดังนี้ พอร์ต Ethernet สำหรับเชื่อมต่อเครือข่าย และพอร์ต USB สำหรับ Export ข้อมูลและอัปเดต Firmware และพอร์ตไฟฟ้า 12V DC

### 3.2 การตรวจสอบสถานะ ZKTeco ประจำสัปดาห์

รายการที่ 1 ตรวจสอบไฟแสดงสถานะที่ตัวเครื่อง ZKTeco ว่าติดสว่างปกติ

รายการที่ 2 ทดสอบการสแกนใบหน้าโดยให้เจ้าหน้าที่ที่ลงทะเบียนไว้สแกนใบหน้าแล้วตรวจสอบว่าระบบจดจำได้ภายในเวลา 2 วินาที

รายการที่ 3 ตรวจสอบ Log ของ Backend ว่าอุปกรณ์ ZKTeco ยังส่งข้อมูลเข้ามาอยู่หรือไม่ โดยใช้คำสั่ง docker logs KMS-backend --tail 50 เพื่อดู Log ล่าสุด 50 บรรทัด

รายการที่ 4 เช็คเวลาของอุปกรณ์ ZKTeco ว่าตรงกับเวลาของ Server หรือไม่ ความคลาดเคลื่อนของเวลาอาจส่งผลต่อการบันทึก Log

### 3.3 การทำความสะอาดกล้องสแกนใบหน้า

ทุก 2 สัปดาห์ ให้ทำความสะอาดกล้องและ Sensor ของ ZKTeco ดังนี้

ขั้นที่ 1 ปิดการทำงานของอุปกรณ์ก่อนทำความสะอาด

ขั้นที่ 2 ใช้ผ้า Microfiber แห้งเช็ดกล้องเบา ๆ ห้ามใช้กระดาษทิชชูเนื่องจากจะทำให้เกิดรอยขีดข่วน

ขั้นที่ 3 หากมีคราบสิ่งสกปรกติดอยู่ ให้ใช้ผ้า Microfiber ชุบน้ำสะอาดเพียงเล็กน้อยแล้วเช็ด ห้ามฉีดน้ำหรือของเหลวโดยตรงที่กล้อง

ขั้นที่ 4 เช็ดหน้าจอ LCD ด้วยผ้า Microfiber แห้ง

### 3.4 การสำรองข้อมูลใบหน้าจาก ZKTeco

ข้อมูลใบหน้าที่บันทึกไว้ใน ZKTeco มีความสำคัญสูงมาก หากเครื่องเสียหายและข้อมูลใบหน้าสูญหาย นักศึกษาและบุคลากรทั้งหมดจะต้องมาลงทะเบียนใหม่

ขั้นที่ 1 เข้าสู่หน้าตั้งค่าของ ZKTeco ผ่านเว็บเบราว์เซอร์โดยพิมพ์ IP ของเครื่องในช่องที่อยู่

ขั้นที่ 2 ล็อกอินด้วยบัญชีผู้ดูแลระบบ

ขั้นที่ 3 ไปที่เมนูสำรองข้อมูลหรือ Backup/Restore แล้วเลือก Export All Data

ขั้นที่ 4 บันทึกไฟล์ที่ Export ได้ไว้ในพื้นที่ปลอดภัยอย่างน้อย 2 ที่ เช่น NAS และ Cloud Storage

ขั้นที่ 5 บันทึกวันที่สำรองไว้ในสมุดบันทึกการบำรุงรักษา

ควรสำรองข้อมูลใบหน้าเดือนละ 1 ครั้ง และทุกครั้งที่มีการเพิ่มหรือลบข้อมูลใบหน้าจำนวนมาก

### 3.5 การอัปเดต Firmware ของ ZKTeco

ก่อนอัปเดต Firmware ให้สำรองข้อมูลทั้งหมดก่อนเสมอ ดาวน์โหลด Firmware เวอร์ชันใหม่จากเว็บไซต์ทางการของ ZKTeco ตรวจสอบให้แน่ใจว่า Firmware เป็นเวอร์ชันที่รองรับรุ่น SmartAC1 โดยเฉพาะ

ขั้นที่ 1 เข้าสู่หน้าตั้งค่าของ ZKTeco ผ่านเว็บเบราว์เซอร์

ขั้นที่ 2 ไปที่เมนูอัปเดต Firmware

ขั้นที่ 3 อัปโหลดไฟล์ Firmware ที่ดาวน์โหลดมาและยืนยัน

ขั้นที่ 4 รอจนกระบวนการอัปเดตเสร็จสมบูรณ์ ห้ามปิดเครื่องระหว่างอัปเดต

ขั้นที่ 5 เครื่องจะรีสตาร์ทอัตโนมัติ รอจนเปิดขึ้นมาแล้วทดสอบการสแกนใบหน้า

## 4. ซอฟต์แวร์: Docker และ Container Management

### 4.1 ภาพรวม Docker Compose

ระบบ KMS ใช้ Docker Compose ในการจัดการ Container ทั้งหมด ไฟล์กำหนดค่าหลักอยู่ที่ docker-compose.yml ซึ่งกำหนด Service ดังนี้

Service postgres เป็น PostgreSQL 15 Alpine รันบน port 34669 มีการตั้งค่า restart always เพื่อให้รีสตาร์ทอัตโนมัติเสมอ

Service redis เป็น Redis Alpine รันบน port 63797 มีการตั้งค่ารหัสผ่านผ่าน Environment Variable และตั้งค่า restart always

Service backend เป็น Node.js Express API Server รันบน port 4556 เชื่อมต่อกับทั้ง postgres และ redis ตั้งค่า restart always

Service frontend เป็น Next.js รันบน port 80 เชื่อมต่อกับ backend ตั้งค่า restart always

### 4.2 คำสั่ง Docker ที่ใช้บ่อยในการดูแลระบบ

การตรวจสอบสถานะ Container ทั้งหมด ให้ใช้คำสั่ง docker compose ps ผลลัพธ์จะแสดงสถานะของแต่ละ Service ถ้า State แสดงว่า running ทุก Service แสดงว่าระบบปกติ

การดู Log ของ Container เฉพาะตัว ให้ใช้คำสั่ง docker logs ชื่อContainer เช่น docker logs KMS-backend เพื่อดู Log ทั้งหมด หรือเพิ่ม --tail 100 เพื่อดูเฉพาะ 100 บรรทัดล่าสุด หรือเพิ่ม -f เพื่อดู Log แบบต่อเนื่อง Real-time

การรีสตาร์ท Container เฉพาะตัว ให้ใช้คำสั่ง docker restart KMS-backend โดยแทนชื่อ Container ตามที่ต้องการ

การรีสตาร์ทระบบทั้งหมดพร้อมกัน ให้ใช้คำสั่ง docker compose down ก่อน แล้วตามด้วย docker compose up -d

การดูการใช้ทรัพยากรของแต่ละ Container ให้ใช้คำสั่ง docker stats เพื่อดูการใช้ CPU RAM และ Network แบบ Real-time

### 4.3 การตรวจสอบ Container ประจำวัน

ทุกวันเช้าก่อนเริ่มทำการ เจ้าหน้าที่ผู้ดูแลระบบควรดำเนินการดังนี้

ขั้นที่ 1 SSH เข้า Server หลักและรันคำสั่ง docker compose ps เพื่อตรวจสอบว่าทุก Container อยู่ในสถานะ running

ขั้นที่ 2 ดู Log ของ Backend ด้วยคำสั่ง docker logs KMS-backend --tail 50 --since 24h เพื่อดู Log ในช่วง 24 ชั่วโมงที่ผ่านมา ตรวจสอบว่าไม่มี Error ที่ผิดปกติ

ขั้นที่ 3 ตรวจสอบพื้นที่ดิสก์ด้วยคำสั่ง df -h เพื่อให้แน่ใจว่าพื้นที่ว่างยังเพียงพอ ควรรักษาพื้นที่ว่างไว้ไม่น้อยกว่า 20 เปอร์เซ็นต์

ขั้นที่ 4 ตรวจสอบการใช้ RAM ด้วยคำสั่ง free -h ให้แน่ใจว่าระบบไม่ใช้ RAM เต็มจนเกินไป

### 4.4 การอัปเดตซอฟต์แวร์

สำหรับ Backend และ Frontend ซึ่งพัฒนาด้วย Node.js และ Next.js การอัปเดตจะดำเนินการผ่าน Docker Compose โดยมีขั้นตอนดังนี้

ขั้นที่ 1 สำรองฐานข้อมูลก่อนอัปเดตเสมอ

ขั้นที่ 2 ดึงโค้ดเวอร์ชันใหม่จาก Git Repository ด้วยคำสั่ง git pull origin main

ขั้นที่ 3 Build Docker Image ใหม่ด้วยคำสั่ง docker compose build

ขั้นที่ 4 รัน Migration ฐานข้อมูลหากมีการเปลี่ยนแปลง Schema โดยใช้คำสั่ง docker compose run backend npx prisma migrate deploy

ขั้นที่ 5 รีสตาร์ท Container ด้วยคำสั่ง docker compose up -d

ขั้นที่ 6 ตรวจสอบ Log เพื่อให้แน่ใจว่า Container ทำงานได้ตามปกติหลังอัปเดต
